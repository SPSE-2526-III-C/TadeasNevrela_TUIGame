# Design Notes

This document preserves the high-level design for the terminal security horror game. Use `plan.md` for implementation order and `memory.md` for decisions that should be retained over time.

## Overview

The game is a terminal-based security operator simulation horror game written in C.

The player does not directly walk through the world. Instead, they operate a facility through a structured security terminal interface with menus, system panels, camera feeds, logs, doors, reports, and integrity monitoring.

Core pillars:

- Cursor-based TUI navigation.
- Menu-driven security OS interface.
- Wolfenstein-style raycasting cameras rendered in ASCII.
- System simulation for doors, logs, alarms, power, and integrity.
- Horror built from system corruption, contradictions, and unreliable reality.

## Player Fantasy

The player is a night security operator monitoring a facility through an old terminal system.

The player can:

- Watch cameras.
- Inspect logs.
- Toggle or inspect doors.
- Monitor system health.
- Respond to warnings.
- Try to infer what is actually happening from unreliable information.

The player cannot simply leave the chair and solve the problem directly.

## Interface Model

The interface behaves like real monitoring software.

Everything is menu-driven:

```text
SECURITY TERMINAL v1.0

> Cameras        <--
  Doors
  Security Logs
  Reports
  Systems
  Emergency Protocols
```

Navigation:

- Up/down moves the cursor.
- Enter selects a menu item.
- Escape or backspace returns to the previous screen.
- Tab may later switch panels.

The MVP should avoid typed commands. Typed commands may become an advanced feature later.

## Core Game Loop

```c
while (running) {
    handle_input();
    update_state();
    generate_events();
    render_ui();
}
```

The player is always acting through the terminal, so all gameplay should route through UI screens and system panels.

## Menu System

The menu hierarchy is tree-like, with stack-based navigation.

Example:

```text
MAIN MENU
├── Cameras
│   ├── CAM 01
│   ├── CAM 02
├── Doors
│   ├── Floor 1
│   ├── Floor 2
├── Security Logs
├── Reports
├── Systems
└── Emergency Protocols
```

The menu itself can become part of the horror. Later corruption may alter labels, move the cursor, hide options, or make selections behave incorrectly.

## Camera System

Each camera is a surveillance feed rendered through a raycasting pseudo-3D view.

Example camera structure:

```c
typedef struct {
    float x;
    float y;
    float angle;
    float fov;
} Camera;
```

Cameras should feel fixed, mechanical, and mediated by the system. The player sees what the terminal claims the camera sees.

## World Representation

The facility is a 2D grid map.

Example tile types:

- `0` empty space.
- `1` wall.
- `2` door.
- `3` anomaly or entity marker.

Example:

```c
#define MAP_W 24
#define MAP_H 24
int map[MAP_H][MAP_W];
```

Use consistent row/column ordering once implementation starts.

## Raycasting

Each camera frame is generated by casting one ray per screen column.

Basic process:

1. Compute ray angle from camera direction and field of view.
2. Step through the map grid until collision.
3. Measure distance to the hit.
4. Convert distance into vertical ASCII wall height.

Core formula:

```c
wall_height = screen_height / distance;
```

The real implementation should guard against division by zero and apply basic correction for fisheye distortion if needed.

## ASCII Camera Rendering

Example:

```text
████████████████
██████    ██████
████        ████
██            ██
```

Visual rules:

- Close objects appear taller and more solid.
- Distant objects appear shorter and thinner.
- Empty space creates gaps.
- Optional fog or brightness falloff can imply distance.

## Door System

Doors are interactive world elements.

States:

- Locked.
- Unlocked.
- Jammed.
- Unknown.

Behavior:

- The player can toggle some doors.
- System events can override door state.
- Some doors may change without player input.
- Door panel state and camera-visible state may later disagree.

## Log System

Security logs show system events in real time.

Example:

```text
[02:14] Motion detected CAM 03
[02:15] Door 2 opened manually
[02:15] No operator detected
```

Horror behaviors:

- Logs contradict camera feeds.
- Logs rewrite themselves.
- Logs reference impossible player actions.
- Logs appear before events happen.

## Systems Panel

The systems panel displays subsystem health:

- Camera network.
- Door control.
- Logging system.
- Power grid.
- External connection.

Important rule:
System status becomes unreliable over time.

## Integrity Instead of Health

There is no player health bar.

The main pressure value is system integrity:

```c
int system_integrity = 100;
```

Integrity loss should affect trust, control, visibility, and system behavior.

## Horror Principles

Horror comes from system failure, not only from monsters.

Core techniques:

- UI inconsistency.
- Contradicting information layers.
- Reality desync between cameras.
- Cursor misalignment.
- Menu restructuring over time.
- Logs that accuse, predict, or deny events.
- Camera feeds that reveal things the map does not confirm.

The player should start by trusting the system, then slowly lose that trust.

## Camera Horror Mechanics

Allowed distortions:

- Impossible geometry.
- Broken perspective rules.
- Flickering entities.
- Missing or extra walls.
- Different cameras showing different realities.
- Entities appearing in some columns but not others.

Example:

- Column 12 shows an entity.
- Column 13 does not.
- Column 14 shows the entity again.

## Design Summary

This is a terminal-based security simulation horror game where:

- Cameras are raycasted pseudo-3D surveillance feeds.
- The UI behaves like real security software.
- The player interacts through structured menus.
- Horror emerges from system corruption and contradictory reality.
- The system itself becomes the antagonist over time.
